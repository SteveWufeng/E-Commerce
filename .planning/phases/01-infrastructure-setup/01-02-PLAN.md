    Write `.github/workflows/deploy.yml`:

    ```yaml
    name: Deploy to Cloudflare Workers

    on:
      push:
        branches: [main]
      workflow_dispatch:  # Allow manual trigger from GitHub UI

    env:
      NODE_VERSION: '20'

    jobs:
      # ─── Build ───────────────────────────────────────
      build:
        name: Build OpenNext Worker
        runs-on: ubuntu-latest
        outputs:
          build-succeeded: ${{ steps.build.outcome }}
        steps:
          - uses: actions/checkout@v4

          - name: Setup Node.js
            uses: actions/setup-node@v4
            with:
              node-version: ${{ env.NODE_VERSION }}
              cache: 'npm'

          - name: Install dependencies
            run: npm ci

          - name: Generate Prisma client
            run: npx prisma generate

          - name: Build with OpenNext
            run: npm run cf:build
            env:
              DATABASE_URL: ${{ secrets.DATABASE_URL }}

          - name: Upload build artifact
            uses: actions/upload-artifact@v4
            with:
              name: opennext-build
              path: |
                .open-next/
                wrangler.toml
                package.json
                package-lock.json
                node_modules/
              retention-days: 1

      # ─── Migrate Database ────────────────────────────
      migrate:
        name: Run Database Migrations
        runs-on: ubuntu-latest
        needs: build
        # Run migrations BEFORE deploy to ensure DB schema matches code
        steps:
          - uses: actions/checkout@v4

          - name: Setup Node.js
            uses: actions/setup-node@v4
            with:
              node-version: ${{ env.NODE_VERSION }}
              cache: 'npm'

          - name: Install dependencies
            run: npm ci

          - name: Run Prisma migrations
            run: npx prisma migrate deploy
            env:
              # GitHub Actions runner HAS TCP access — direct Supabase connection works here
              DATABASE_URL: ${{ secrets.DATABASE_URL }}

      # ─── Deploy ──────────────────────────────────────
      deploy:
        name: Deploy to Cloudflare Workers
        runs-on: ubuntu-latest
        needs: [build, migrate]
        if: success()
        steps:
          - name: Download build artifact
            uses: actions/download-artifact@v4
            with:
              name: opennext-build

          - name: Deploy to Cloudflare Workers
            uses: cloudflare/wrangler-action@v3
            with:
              apiToken: ${{ secrets.CLOUDFLARE_API_TOKEN }}
              accountId: ${{ secrets.CLOUDFLARE_ACCOUNT_ID }}
              command: deploy
            env:
              # Secrets are injected into Workers runtime per D-09/D-10
              DATABASE_URL: ${{ secrets.DATABASE_URL }}
              NEXTAUTH_SECRET: ${{ secrets.NEXTAUTH_SECRET }}
    ```

    Key design decisions:
    - **Three jobs: build → migrate → deploy** — Migrations run from GitHub Actions runner (which HAS TCP access to PostgreSQL), NOT from Workers runtime (which does NOT have TCP access). This is the only viable approach without a connection proxy.
    - **wrangler-action@v3** — Official Cloudflare GitHub Action. It handles authentication via `CLOUDFLARE_API_TOKEN` and `CLOUDFLARE_ACCOUNT_ID` secrets.
    - **DATABASE_URL in build step** — Required for Prisma generate (reads schema.prisma). At build time, Prisma only needs the schema, not a live connection, but env var must be present for `prisma generate` to succeed.
    - **artifact upload/download** — Preserves the OpenNext build output between jobs. Retention set to 1 day to minimize storage cost.
    - **workflow_dispatch** — Allows manual deploys from GitHub UI for testing.

    Note on Cloudflare API Token:
    - User must create a Cloudflare API Token with "Edit Cloudflare Workers" permission at: https://dash.cloudflare.com/profile/api-tokens
    - Token goes into GitHub repo secrets as `CLOUDFLARE_API_TOKEN`
    - Account ID found in Cloudflare dashboard → Workers & Pages → Overview
    - Account ID goes into GitHub repo secrets as `CLOUDFLARE_ACCOUNT_ID`
  </action>

    This task documents the migration strategy and ensures the schema is production-ready. No structural changes to `prisma/schema.prisma` are needed — the existing schema is complete and correct.

    ### Migration Strategy for Cloudflare Workers

    **Why migrations must run in CI/CD, not Workers runtime:**
    Cloudflare Workers are V8 isolates — they have NO raw TCP socket access. Prisma cannot open PostgreSQL connections from Workers. The only place with TCP access is the GitHub Actions runner (Linux VM).

    **Strategy:**
    1. Developer creates migration locally: `npx prisma migrate dev --name <name>`
    2. Commit migration files (`prisma/migrations/`) to git
    3. On push to main, CI/CD runs `npx prisma migrate deploy` BEFORE deploying the Worker
    4. The deployed Worker code is always paired with a migrated database

    **For runtime database queries from Workers:**
    To enable Prisma queries FROM the Cloudflare Workers runtime, we need a connection proxy. Options:
    - **Cloudflare Hyperdrive** (recommended for MVP): Free tier, 10GB/month. Add to `wrangler.toml`:
      ```toml
      [[hyperdrive]]
      binding = "HYPERDRIVE"
      id = "<hyperdrive-configuration-id>"
      ```
      Then update `DATABASE_URL` in Workers to point to Hyperdrive binding.
    
    - **Prisma Accelerate**: $5/month, managed connection pool with edge support. Update `prisma/schema.prisma` datasource:
      ```prisma
      datasource db {
        provider = "postgresql"
        url      = env("DATABASE_URL")
        directUrl = env("DIRECT_URL")  // for migrations only
      }
      ```
    
    Currently D-07 says "direct connection, no PgBouncer" — but this is physically impossible from Workers without a proxy. This is a **known constraint** that will be resolved during execution. The executor should:
    1. Set up Hyperdrive (free, low friction)
    2. Update DATABASE_URL in wrangler secrets to the Hyperdrive binding
    3. Document the change in SUMMARY.md

## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| GitHub Actions runner → Supabase PostgreSQL | Database migrations run over public internet |
| GitHub Actions runner → Cloudflare API | Deploy credentials cross trust boundary |
| Cloudflare Workers → Supabase PostgreSQL | Runtime queries cross public internet (requires Hyperdrive proxy) |
| Developer machine → GitHub repo | Source code and migration files pushed to cloud |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-01-05 | Spoofing | CI/CD pipeline | mitigate | Cloudflare API token scoped to Workers edit only (not account-wide). Token stored in GitHub encrypted secrets. |
| T-01-06 | Tampering | Migration job | mitigate | Prisma migrate deploy is idempotent — only applies unapplied migrations. Migration files are version-controlled and reviewed. |
